# Архитектура приложения

## 1. Общая архитектура

Приложение построено на микросервисной архитектуре и состоит из нескольких независимых сервисов, каждый из которых отвечает за свою бизнес-логику.

### Выделенные сервисы:

- **gateway-service** — единая точка входа (API Gateway). Принимает все внешние запросы и маршрутизирует их к соответствующим сервисам.
- **event-service** — управление событиями (создание, обновление, поиск).
- **request-service** — управление заявками на участие в событиях.
- **user-service** — управление пользователями.
- **category-service** — управление категориями.
- **stats-server** — сбор и предоставление статистики.
- **config-server** — централизованное хранилище конфигураций.
- **eureka-server** — сервис регистрации и обнаружения сервисов.

---

## 2. Взаимодействие сервисов

### Внешние запросы

Все внешние запросы поступают через **gateway-service**.

Gateway:
- определяет маршрут на основе URL
- перенаправляет запрос в нужный сервис через **Eureka (service discovery)**

Пример:

`/admin/events → gateway → event-service`

---

### Внутреннее взаимодействие

Сервисы взаимодействуют друг с другом напрямую:

- через **Feign-клиенты**
- через HTTP REST вызовы

Примеры:

- **event-service → request-service** — получение количества подтверждённых заявок
- **event-service → user-service** — получение информации о пользователе
- **event-service → category-service** — получение информации о категории
- **event-service → stats-server** — получение статистики просмотров

---

## 3. Конфигурация

Конфигурации сервисов вынесены в **config-server**.

### Где находятся конфигурации:

- Все настройки (БД, логирование, маршруты gateway и т.д.) лежат во внешнем репозитории конфигураций
- Сервисы получают конфигурацию при запуске через:

`spring.config.import=configserver:`

---

### Примеры конфигураций:

- datasource (PostgreSQL)
- logging.level
- gateway routes
- feign настройки
- resilience4j (circuit breaker)

---

## 4. Внутренний API

Внутренний API — это REST API, который сервисы используют для общения друг с другом.

### Основные взаимодействия:

- **request-service**
  - получение количества подтверждённых заявок по eventId

- **user-service**
  - получение данных пользователя по id

- **category-service**
  - получение категории по id

- **stats-server**
  - получение количества просмотров событий

Все вызовы выполняются через Feign-клиенты.

---

## 5. Внешний API

Внешний API доступен через **gateway-service**.

Он описан в спецификации (Swagger / OpenAPI), которая используется в групповом проекте.

### Основные группы эндпоинтов:

- `/admin/**` — административные операции
- `/users/**` — действия пользователей
- `/events/**` — публичные события
- `/categories/**` — категории
- `/compilations/**` — подборки событий

---

## 6. Итог

- Gateway — точка входа
- Eureka — сервис discovery
- Config Server — централизованная конфигурация
- Остальные сервисы — бизнес-логика
- Взаимодействие — REST + Feign

Архитектура обеспечивает слабую связанность сервисов и масштабируемость системы.
